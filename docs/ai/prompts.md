カスタムプロンプト一覧
=========================

概要
-------------------------

このドキュメントは、ワークスペースで利用できるカスタムプロンプトを用途別に整理した一覧です。
各プロンプトの役割、入力の与え方、想定出力をひと目で確認できるようにし、依頼内容に合うプロンプト選定を支援します。
プレフィックスと用途の組み合わせを参考に、呼び出し候補を素早く絞り込めます。

使い分けの指針
-------------------------

### doc-系とgenerate-系の選択

ドキュメント生成(`doc-`)はプロジェクト成果物や仕様書の出力が目的です。
カスタマイズファイルの新規作成(`generate-agent`・`generate-prompt`・`generate-skill`・`generate-instructions`)は
AIエージェント運用基盤の整備が目的です。混同しやすい場合は主な出力の種類で区別できます。

### チェック系の選択基準

コードの品質評価には`check-code`、Markdownの記法・表記違反検出には`check-markdown`を選びます。
読者適合性(用語・表現レベル)の評価は`check-markdown`完了後に`check-document`で行います。
プロンプト・スキル・エージェント・インストラクションの各ファイル品質は対応する`check-`プロンプトで個別に確認します。

### 図表形式の選択基準

D2形式(テキストベースでGit差分管理したい場合)は`d2-`プレフィックスのプロンプトを使います。
Draw.io形式(ER図・スイムレーン図を視覚的に編集したい場合)は`drawio-`プレフィックスを使います。
フロー・シーケンス・アクティビティの違いは図の種類で選択します。

### セキュリティ対応の流れ

依存関係全体のスキャンには`scan-vulnerability`を使い、出力されたJSONを`report-vulnerability`でトリアージします。
個別CVEの詳細調査はcve-analyzerスキルと組み合わせて行います。

### 要件定義から設計への流れ

`doc-problem` → `doc-proposal` → `doc-requirements` → `doc-rest_api` / `design-database`の順に進めると、
問題分析・改善提案・要件定義・設計の一貫したドキュメントチェーンを構築できます。

効率化のポイント
-------------------------

### プレフィックスで候補を絞る

- `doc-` : プロジェクト文書の生成・更新
- `design-` : 設計関連ドキュメントの生成・更新
- `check-` : 成果物の品質チェック・レビュー
- `d2-` : D2言語形式の図表生成
- `drawio-` : Draw.io XML形式の図表生成
- `generate-` : AIエージェント運用基盤ファイルの新規作成
- `task-` : タスク管理・変換・可視化
- `infra-` : インフラ構成・運用ファイルの生成
- `testcase-` : テストデータ・テスト設計ファイルの生成
- `issue-` : チケット・課題管理操作
- `proposal-` : 改善提案の評価・比較資料作成
- `scan-` / `report-` : 脆弱性スキャンとトリアージレポート生成

プレフィックスを持たないプロンプト(`agent-guide`・`extract-urls`・`workplan`・`checklist`)は、
固有用途のユーティリティとして個別に選択します。

### 前後関係のある組み合わせ活用

- `checklist` でチケット分析 → `task-generate_tasklist` でタスク分解 → `task-generate_issue` でissueたたき台作成
- `generate-prompt` で新規プロンプト作成 → `check-prompt` で品質確認
- `design-ui_layout` でUI構成ルール確定 → `doc-ui_pattern` で画面単位の仕様書作成
- `infra-hearing` でインフラ要件を構造化 → `infra-terraform` でコード生成
- `scan-vulnerability` で依存関係を検査 → `report-vulnerability` で優先度付き対応方針を作成

### 固定エージェントの確認

YAMLフロントマターに`agent`が指定されているプロンプトは、専門エージェントが固定で起動します。
指定がないプロンプト(`check-code`・`annotate-code`等)は、ファイル拡張子に応じた言語別エージェントが動的に選択されます。

各プロンプトの固定エージェントは、各プロンプトファイルのYAMLフロントマター`agent`フィールドで確認できます。

まとめ
-------------------------

カスタムプロンプトは、用途・入力・出力の3要素で整理すると選びやすくなります。
プレフィックスを起点に候補を絞り、できることと主な出力で最終確認することで呼び出しミスを防げます。
前後関係のある組み合わせを把握しておくと、一連のドキュメント作成作業を効率よく進められます。
